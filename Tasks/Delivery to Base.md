# Доставка брейнрота на базу

## Цель

Замкнуть основной игровой цикл из [Main Game Design.md](../Game Design/Main Game Design.md):

1. ~~**Собрать** брейнрота в лабиринте~~ ✓ (готово, см. `BrainrotPickup` + `BrainrotInput`)
2. **Принести** его на свою базу ← *эта задача*
3. **Получать** пассивный доход от размещённых на базе брейнротов ← *эта задача*

Без доставки текущий пикап — это игрушка-однодневка: подобрал и положил обратно. После доставки появляется смысл бегать в лабиринт.

## Функциональные требования

1. **Триггер доставки** — игрок, несущий брейнрота, входит в зону **своей** базы → брейнрот размещается на свободном слоте, перестаёт быть в инвентаре игрока.
2. **Пассивный доход** — каждый размещённый брейнрот приносит **N мон/сек** в зависимости от редкости (см. таблицу в [Brainrots.md](../Game Design/Brainrots.md) и [Economy.md § 1](../Game Design/Economy.md)):
   - Common: 1, Rare: 4, Epic: 15, Legendary: 60, Mythic: 250, Gogly: 1000
3. **Валюта** — копится у игрока, видна через стандартный Roblox `leaderstats` (автоматический UI в правом верхнем углу).
4. **Лимит вместимости базы** — стартово **4 слота** (lvl 1 по [Economy.md § 4](../Game Design/Economy.md)). Если все заняты — доставка отклоняется, брейнрот остаётся над головой.
5. **Вместимость переноски** — игрок одновременно несёт **до 2 брейнротов** (текущий код в `BrainrotPickup` ограничивает 1 — нужно править).
6. **Ownership** — чужие игроки видят размещённых брейнротов на чужих базах, но не могут с ними взаимодействовать.
7. **Сохранение между сессиями** — вне скоупа этой задачи.

## Решённые вопросы

1. **`BaseManager` ownership** — прочитать скрипт в начале реализации; если привязка уже есть — переиспользовать, если нет — сделать.
2. **Редкость брейнрота** — хранилища нет. Заводим **атрибут `Rarity`** (значения: `Common` / `Rare` / `Epic` / `Legendary` / `Mythic` / `Gogly`) на каждой модели. Доход берём из ModuleScript-таблицы `Rarity → income/sec`.
   - Существующим 6 моделям проставляем атрибут одноразовым скриптом (через `execute_luau`). Маппинг — best-effort по имени; неоднозначные дефолтятся в `Common`, дальше пользователь правит вручную.
3. **Зона доставки** — **вся база как зона**. Слушаем `Touched` на любом `BasePart` внутри `bazapl*`, фильтруем по владельцу.
4. **Визуальное размещение** — **произвольные позиции** на базе. Берём случайную точку в bounding box базы + небольшой Y-offset, чтоб не клипалось в пол.
5. **Вместимость переноски — 2** одновременно. **Меняем `BrainrotPickup`**: `carrying[player]` становится массивом max-length 2; ProximityPrompt пикапа отключается, только когда массив полон.
6. **UI валюты — стандартный `leaderstats`** (Roblox сам отрисует в правом верхнем углу).
7. **Смерть с брейнротом** — брейнрот **уничтожается** (`:Destroy()`). **Меняем `BrainrotPickup.hum.Died` handler**: вместо `drop()` — destroy всех несомых.

### Доработка пикапа (под новые вводные)
- [ ] **Разрешить нести 2 одновременно**: в `BrainrotPickup` сделать `carrying[player]` массивом, лимит 2. Над головой класть стопкой (offset по Y).
- [ ] **При смерти — уничтожать**, не дропать: в `setupCharacter` `hum.Died` заменить `drop(player)` на цикл `:Destroy()` всех несомых + сброс `carrying[player]` и атрибута `CarryingBrainrot`.

### Ownership базы
- [ ] Если `BaseManager` ещё не привязывает — реализовать: на `Players.PlayerAdded` назначать свободный `bazapl1-4`, записать `Owner` (атрибут с UserId) на саму базу.
- [ ] На `PlayerRemoving` — освобождать (брейнроты на базе остаются, доход капает в никуда — see Edge cases).

### Доставка
- [ ] В новом скрипте `BrainrotDelivery` (Server, ServerScriptService) подписаться на `Touched` всех `BasePart`-потомков каждой `bazapl*`:
   - получить игрока из коснувшейся модели;
   - сверить с `Owner` базы — если чужой, игнор;
   - если у игрока есть несомые брейнроты и есть свободный слот — разместить **первого** из массива.
- [ ] Размещение: вынуть брейнрота из массива переноски, открепить от анкора, поставить **в произвольную точку в bounding box базы** + небольшой Y-offset, заанкорить, добавить в массив `placed[base]`.

### Доход (leaderstats + income tick)
- [ ] На `PlayerAdded`: создать `Folder leaderstats` в игроке, в нём `IntValue Coins = 0`.
- [ ] Один общий `task.spawn` цикл с `task.wait(1)`: для каждого игрока суммировать `BrainrotConfig.income[Rarity]` по всем брейнротам на его базе, прибавлять к `Coins`.

### Ownership-защита и лимит
- [ ] На размещённых брейнротах ставить `ProximityPrompt.Enabled = false` (брейнрот заперт на базе).
- [ ] Атрибут `Capacity = 4` на базе. При попытке доставки если `#placed >= Capacity` — отклонить, оставить в руках, послать клиенту сообщение через RemoteEvent (новый `DeliveryRejected` или переиспользовать `BrainrotAtBase` со статусом).

## Граничные случаи

- **Игрок несёт брейнрота, заходит на чужую базу** — игнорировать (триггер срабатывает только на своей, фильтр по `Owner`).
- **База заполнена** — отказ доставки + сообщение клиенту, брейнрот остаётся над головой. Игрок может стоять на базе с брейнротом и не размещать — это ок.
- **Игрок несёт 2 брейнрота, на базе свободен 1 слот** — разместить только первого, второй остаётся над головой.
- **Игрок дисконнектится с расставленными брейнротами** — оставить на базе. Доход больше не капает (нет `Coins` для отсутствующего игрока). Офлайн-доход — отдельная задача в backlog.
- **Игрок умирает с брейнротами над головой** — все несомые уничтожаются (по решению #7). Размещённые на базе остаются нетронутыми.
- **Два игрока заходят в одну `bazapl*` зону одновременно** — фильтр по `Owner` решает: только владелец инициирует доставку.

## Acceptance criteria

Задача закрыта, когда:

1. ✅ Я могу подобрать **2 брейнрота** одновременно (стопкой над головой).
2. ✅ Подобрал Common, принёс на свою базу — автоматически размещается на свободной точке базы.
3. ✅ В `leaderstats` появилось `Coins`, идёт +1 мон/сек.
4. ✅ Положил рядом второго Common — стало +2 мон/сек.
5. ✅ Положил Mythic — стало +252 мон/сек.
6. ✅ Другой игрок (тестбот) подходит к моей базе и не может поднять моих брейнротов.
7. ✅ После 4 размещённых (стартовая вместимость) пятый не размещается — брейнрот остаётся в руках.
8. ✅ Умер с брейнротами над головой — они исчезли, размещённые на базе не пострадали.

## Что ЯВНО вне скоупа этой задачи

- Апгрейды (скорость, вместимость переноски, вместимость базы) — отдельные задачи [магазин].
- Сохранение прогресса между сессиями (DataStore).
- Спавнер брейнротов в лабиринтах — отдельная задача.
- Офлайн-доход.
- Защита от чита (`Coins` на сервере достаточно для MVP, но клиент не должен иметь право менять).
