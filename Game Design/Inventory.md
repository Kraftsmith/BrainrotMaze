# Инвентарь

Точка, где игрок видит, что у него стоит на базе. Это «бухгалтерия» прогресса: какие брейнроты собраны, какие тиры закрыты, и сколько они в сумме приносят. Не путать с **carry-стеком** (тем, что несёшь над головой) — он живёт в HUD ([Main Game Design.md](Main%20Game%20Design.md) § Взятие брейнрота) и в инвентарь не показывается.

## Цель и scope

- **MVP:** показать игроку список placed-брейнротов на его базе с группировкой по имени и доходом.
- Не заменяет HUD-индикатор carry над головой.
- Не позволяет ничего перетаскивать, продавать или удалять (это идеи на будущее, см. § Идеи).

## Доступ

- **Кнопка `🗂 Inventory`** — постоянно видна в правом верхнем углу, под кнопкой Shop'а (одинаковый стиль).
- **Клавиша `I`** — открывает/закрывает (одинаково).
- Доступен **везде**: на базе, в лабиринте, на spawn-platform. Можно проверить «сколько у меня сейчас приносит» хоть посреди рейда.
- Не блокирует движение/бой, оверлей.

```
│   Lirilì Larilà            × 4   +100/s │
│   Pipi Kiwi                × 2    +20/s │
└─────────────────────────────────────────┘
```

- **Шапка**: `<placed> / <capacity> slots · +<sum> coins/sec total`. Capacity берётся из атрибута `Capacity` базы игрока (текущий лимит после апгрейдов в магазине). Сумма — то же, что считает `BrainrotDelivery.income tick`.
- **Группы по rarity**: Mythic → Legendary → Epic → Rare → Common (Gogly выше Mythic, см. таблицу ниже). Каждый rarity-заголовок с цветной плашкой соответствующего тира. Группы с 0 элементов **скрыты** (без пустых заголовков).
- **Внутри группы — сортировка по убыванию дохода/шт** (т.е. сначала самые ценные в тире). При равном доходе — алфавит.
- **Row**: `<name>  × <count>   +<row_income>/s`, где `row_income = count × per_brainrot_income × vipMul`. VIP-множитель учитываем (см. [Base.md § VIP Base](Base.md)).
- **Иконок/3D-превью нет** в MVP. Цвет rarity достаточно различает тиры. Превью — § Идеи.
- **Полоса прокрутки** появляется при cap=30 заполненной разными брейнротами (~6+ разных видов). Использовать `ScrollingFrame` с автоматическим CanvasSize.

## Поведение

1. **Открытие** — мгновенно; рендер из текущего состояния, без round-trip к серверу. Источник истины — теги/атрибуты на моделях в Workspace, всё это уже реплицировано клиенту.
2. **Live-обновление** (только пока инвентарь открыт):
   - Новый брейнрот сдан → `CollectionService:GetInstanceAddedSignal("PlacedBrainrot")` → re-render.
   - Брейнрот разрушен/убран с базы → `GetInstanceRemovedSignal("PlacedBrainrot")` или `model.AncestryChanged` → re-render.
   - `Capacity` базы изменился (после покупки слота) → re-render шапки.
   - `Vip` базы изменился → re-render income-чисел.
   - Дребезг: ререндер дебаунсим в один кадр (`task.defer`), чтобы 5 идущих подряд rehydration-сигналов не вызвали 5 ререндеров.
3. **При закрытом инвентаре** — все подписки висят, но re-render no-op. Дёшево.
4. **При первом заходе** на пустую базу — `Empty. Place a brainrot to start earning.` (один центрированный текст вместо списка).
5. **Чужие базы.** Видим только СВОЙ инвентарь. Открыв инвентарь на чужой базе, игрок видит свои брейнроты на своей базе (а не на той, рядом с которой стоит). Фильтр — `model:GetAttribute("PlacedBy") == LocalPlayer.Name`.

## Источник данных и архитектура

Минимизируем new code. Используем существующий контракт:

- **Тег `PlacedBrainrot`** на модели — единственный признак «это брейнрот на базе».
- Атрибут `PlacedBy` на модели = ник владельца. Источник истины — `BrainrotLifecycle.transition(model, Placed, {player})` (см. [`BrainrotLifecycle.lua`](../src/ServerStorage/BrainrotLifecycle.lua)).
- Атрибут `Rarity` на модели = тир. Унаследован от шаблона.
- Атрибут `Capacity` на базе игрока. База игрока находится через `bazapl1..4` с `Owner == LocalPlayer.Name`.
- `BrainrotConfig.getIncome(model)` — модуль в `ReplicatedStorage`, доступен клиенту. Учитывает имя и rarity-fallback.

Ничего нового на сервере **не нужно**. Нет RemoteEvent'ов, нет дополнительных атрибутов, нет новых полей в `PlayerData`. Клиентский скрипт читает теги и атрибуты напрямую, потому что они реплицируются автоматически.

## UX правила

1. **Тоггл-кнопка всегда видна** (как Shop). Не прячется когда инвентарь открыт.
2. **Не закрывает основной HUD** — coins, carry-индикатор остаются видны (инвентарь Размер ~420×500, занимает центр-правую часть экрана).
3. **Нумерация в шапке live**: счётчик `placed` обновляется при каждом изменении, даже до полного ререндера списка (отдельная подписка-light).
4. **Цвет rarity-плашки на ховере row'а** — слегка светлее (UIStroke 1px того же цвета). Кликабельность row'а не подразумевается (ничего не делает на клик в MVP).

## Что НЕ в инвентаре (отдельные системы)

- **Carry-стек** (брейнроты над головой) — HUD-индикатор атрибута `CarryingBrainrot` (см. `BrainrotInput.client.lua`). В инвентаре их **не дублируем** — иначе путаница «у меня 5? или это carrying + placed?».
- **Покупки и апгрейды** — отдельный экран `Shop` ([Shop.md](Shop.md)).
- **Прогресс по лабиринтам / разблокировка** — отдельный экран ([Labyrinths.md](Labyrinths.md)).
- **Лидерборд / рейтинг** — не в дизайне MVP.

## Технические артефакты

- `StarterPlayerScripts.InventoryClient` (новый LocalScript) — UI + подписки на CollectionService и атрибуты. Стиль и компоновка — копия `ShopClient.client.lua` (ScreenGui + Frame + ScrollingFrame).
- `ReplicatedStorage.BrainrotConfig` — добавить таблицу `BrainrotConfig.rarityColor[<rarity>] = Color3` (и tier-order для сортировки групп). Это переиспользуется как для инвентаря, так и для будущих rarity-меток над брейнротами в лабиринте.
- Сервер — без правок.

### Расчёт total income (для шапки)

```
total = 0
для каждого M в CollectionService:GetTagged("PlacedBrainrot"):
    если M:GetAttribute("PlacedBy") == LocalPlayer.Name и M.Parent ~= nil:
        base = M.Parent
        vipMul = (base:GetAttribute("Vip") == true) ? 2 : 1
        total += BrainrotConfig.getIncome(M) * vipMul
```

Должно совпадать с серверным `BrainrotDelivery.income tick`. Если не совпадает — у нас дрейф между server-state и client-snapshot, надо чинить (но в норме всё реплицируется атомарно).

### Поиск своей базы (для Capacity)

```lua
for _, name in {"bazapl1","bazapl2","bazapl3","bazapl4"} do
    local b = workspace:FindFirstChild(name)
    if b and b:GetAttribute("Owner") == player.Name then
        return b
    end
end
```

Если базы нет (BaseManager ещё не назначил) — шапка показывает `0 / ?? slots`, ререндерим по `GetAttributeChangedSignal("Owner")` на каждой `bazapl_`.

## Идеи на будущее (не для MVP)

- **3D-превью**: `ViewportFrame` с миниатюрой модели. Но шаблоны весят (десятки parts), 5–8 одновременных ViewportFrame'ов могут просесть FPS на слабых машинах. Сначала замерим, потом включим.
- **Click-to-locate**: клик по row → камера наводится на конкретный экземпляр на базе (если их несколько — на первый по `SlotIndex`).
- **Sell from inventory**: продать брейнрота прямо из инвентаря за процент от income (например, 1 час дохода). Спорный, ломает прогрессию — обсуждать отдельно.
- **Drag-reorder**: двигать брейнрота в другой `SlotIndex`. Сейчас слоты автоматические по порядку прихода.
- **Filter / search**: фильтр по rarity, поиск по имени. Имеет смысл при cap=30 и большом разнообразии.
- **Compare** на ховере: «при покупке слота получишь возможность поставить ещё одного — лучшее, что у тебя есть, даст +N».
- **Diff highlight**: новые брейнроты с прошлого открытия инвентаря — подсвечены 5 секунд после открытия.
- **Rarity-bar на брейнроте в лабиринте** (тот же `BrainrotConfig.rarityColor`, который заведём здесь): цветная подсветка над прокси-промптом, чтобы издалека видеть, что там за тир.
